# WireGuard Mesh Manager — Deployment Topology

This document describes the recommended deployment topology for WireGuard Mesh Manager (WMM) in production environments, covering TLS termination, reverse proxy configuration, trusted proxy setup, and file permissions.

## Architecture Overview

```
Internet → TLS Terminator → Reverse Proxy → WMM Backend → Database
                     ↓
                 WMM Frontend
```

**Important**: WMM manages WireGuard configurations using a **mesh topology** where all devices connect directly to each other. There is **no central WireGuard server** component that needs to be deployed or managed. The WMM backend service handles configuration generation, key management, and the admin API only.

For detailed information about mesh topology architecture, including endpoint selection rules, deployment patterns, and implementation details, see [Mesh Topology Architecture](MESH_TOPOLOGY_ARCHITECTURE.md).

## Deployment Options

### Option 1: Reverse Proxy with TLS Termination (Recommended)

**Components:**

- **TLS Terminator**: Nginx, Caddy, or cloud load balancer
- **Reverse Proxy**: Handles routing, rate limiting, and security headers
- **WMM Backend**: Python FastAPI service for config generation and management API (internal only)
- **WMM Frontend**: Next.js application for admin interface (internal only)
- **Database**: SQLite with proper file permissions
- **WireGuard Devices**: The actual WireGuard peers that run on your servers/clients (not managed by WMM)

**Note on WireGuard deployment**: WMM generates configurations but does not run WireGuard itself. Each device runs its own WireGuard instance using the generated configuration. This enables full mesh connectivity without requiring a central WireGuard server.

### Option 2: Cloud Native Deployment

**Components:**

- **Cloud Load Balancer**: AWS ALB, GCP L7 LB, or Azure Application Gateway
- **Kubernetes/Containers**: Orchestrated WMM services (backend + frontend)
- **Managed Database**: Optional migration from SQLite to PostgreSQL
- **Secret Management**: HashiCorp Vault or cloud KMS

**Mesh topology considerations**: In cloud environments, devices can be deployed across multiple regions or availability zones. Each device runs WireGuard locally using configurations generated by WMM, establishing direct tunnels with peers regardless of their cloud provider or region.

## TLS Configuration

### TLS Termination Requirements

1. **TLS Version**: Minimum TLS 1.2, preferably TLS 1.3
2. **Certificates**: Use valid certificates from a trusted CA
3. **HSTS**: Enable HTTP Strict Transport Security
4. **OCSP Stapling**: Enable for certificate validation

### Recommended Caddy Configuration

```caddyfile
# /etc/caddy/Caddyfile
wireguard.example.com {
    tls {
        protocols tls1.2 tls1.3
        ciphers TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384 TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
    }

    handle_path /api/* {
        reverse_proxy localhost:8000
        header_up X-Forwarded-Proto {scheme}
        header_up X-Forwarded-Host {host}
        header_up X-Forwarded-For {remote}
        header_up X-Real-IP {remote}
    }

    handle {
        reverse_proxy localhost:3000
        header_up X-Forwarded-Proto {scheme}
        header_up X-Forwarded-Host {host}
    }

    # Security headers
    header {
        Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"
        X-Frame-Options "DENY"
        X-Content-Type-Options "nosniff"
        X-XSS-Protection "1; mode=block"
        Referrer-Policy "strict-origin-when-cross-origin"
        Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline';"
    }
}
```

### Recommended Nginx Configuration

```nginx
# /etc/nginx/sites-available/wireguard-mesh-manager
server {
    listen 443 ssl http2;
    server_name wireguard.example.com;

    # TLS Configuration
    ssl_certificate /etc/ssl/certs/wireguard.crt;
    ssl_certificate_key /etc/ssl/private/wireguard.key;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-RSA-AES256-GCM-SHA512:DHE-RSA-AES256-GCM-SHA512;
    ssl_prefer_server_ciphers off;
    ssl_session_cache shared:SSL:10m;

    # Security Headers
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
    add_header X-Frame-Options "DENY" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # Backend API
    location /api/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header X-Forwarded-Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Real-IP $remote_addr;

        # Rate limiting
        limit_req zone=api burst=10 nodelay;
    }

    # Frontend
    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header X-Forwarded-Host $host;
    }
}

# Rate limiting
http {
    limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;
    limit_req_zone $binary_remote_addr zone=device_api:10m rate=5r/s;
}
```

## Mesh Topology Deployment Considerations

This section covers deployment considerations specific to mesh topology. For comprehensive documentation on mesh topology architecture, including endpoint selection rules, deployment patterns, and detailed implementation information, see [Mesh Topology Architecture](MESH_TOPOLOGY_ARCHITECTURE.md).

### Key Differences from Traditional WireGuard Deployments

Traditional WireGuard deployments often use a hub-and-spoke model with a central server that all clients connect to. WMM uses a different approach:

- **No central WireGuard server**: Each device runs WireGuard independently
- **Full mesh connectivity**: Devices establish direct tunnels with all other devices
- **WMM as configuration manager**: WMM generates, distributes, and manages configurations, but doesn't participate in the WireGuard network itself
- **No WireGuard process on WMM backend**: The backend service only handles HTTP requests for config generation and management

### Network Architecture

```
[Device A] --<--> [Device B] --<--> [Device C]
    ^                ^                 ^
    |________________|_________________|
                      |
                 (Full Mesh)
                      |
                  [WMM Backend]
               (Config Management)
```

Each device runs its own WireGuard interface with peers configured by WMM:

1. **Device A** connects to Device B and Device C directly
2. **Device B** connects to Device A and Device C directly
3. **Device C** connects to Device A and Device B directly

WMM's role is limited to:
- Generating device configurations
- Managing encryption keys
- Providing configuration download API for devices
- Admin interface for network management

### Firewall and Port Requirements

For mesh topology to work correctly, each device's firewall must allow:

- **Inbound UDP on WireGuard port** (default 51820) for all expected peer connections
- **Outbound UDP on WireGuard port** for initiating connections to peers
- **NAT traversal**: Devices behind NAT must have port forwarding configured or use NAT traversal techniques

Example firewall rules for a Linux device:

```bash
# Allow WireGuard traffic (both inbound and outbound)
sudo ufw allow 51820/udp comment 'WireGuard VPN'

# Or with iptables
sudo iptables -A INPUT -p udp --dport 51820 -j ACCEPT
sudo iptables -A OUTPUT -p udp --dport 51820 -j ACCEPT
```

### Device Deployment Process

1. **Configure device in WMM**: Add device with internal and external endpoints
2. **Generate keys**: Create WireGuard key pair for the device
3. **Export configuration**: Download device configuration from WMM
4. **Deploy to device**: Apply configuration to the device's WireGuard instance
5. **Verify connectivity**: Test connections between devices

Example device deployment on Linux:

```bash
# Download configuration from WMM
curl -H "Authorization: Bearer <device-api-key>" \
  https://wmm.example.com/api/devices/<device-id>/config > wg0.conf

# Apply configuration
sudo cp wg0.conf /etc/wireguard/
sudo wg-quick up wg0

# Verify connectivity
sudo wg show
```

### High Availability Considerations

Since there is no central server, the mesh topology provides inherent high availability:

- **No single point of failure**: If one device goes down, others can still communicate
- **Redundant paths**: Multiple routing paths available in larger networks
- **Graceful degradation**: Partial network failures don't affect unrelated device pairs

For WMM itself (the management service), follow standard high availability practices:

- Deploy multiple backend instances behind a load balancer
- Use database replication or managed database services
- Configure health checks and automatic failover

## Reverse Proxy Configuration

### Routing Rules

1. **API Routes**: `/api/*` → Backend (localhost:8000)
2. **Device Config Retrieval**: `/api/devices/*/config` → Special rate limiting
3. **Frontend Routes**: All other paths → Frontend (localhost:3000)
4. **Health Checks**: `/health` → Backend health endpoint

### Rate Limiting Strategy

```nginx
# Different rate limits for different endpoints
location /api/devices/*/config {
    limit_req zone=device_api burst=5 nodelay;
    proxy_pass http://127.0.0.1:8000;
}

location /api/ {
    limit_req zone=api burst=10 nodelay;
    proxy_pass http://127.0.0.1:8000;
}
```

## Trusted Proxy Configuration

### Backend Configuration

The backend must be configured to trust the reverse proxy for proper IP address handling:

```bash
# Environment variables for backend
TRUSTED_PROXY_HEADERS=x-forwarded-for,x-forwarded-proto
TRUSTED_PROXY_COUNT=1  # Number of trusted proxies
```

### Proxy Headers to Forward

- `X-Forwarded-For`: Original client IP address
- `X-Forwarded-Proto`: Original protocol (http/https)
- `X-Forwarded-Host`: Original host header
- `X-Real-IP`: Client IP (alternative to X-Forwarded-For)

### Cloud Provider Considerations

**AWS ALB/ELB:**

```bash
TRUSTED_PROXY_COUNT=1
TRUSTED_PROXY_HEADERS=x-forwarded-for,x-forwarded-proto
```

**Google Cloud Load Balancer:**

```bash
TRUSTED_PROXY_COUNT=2  # GCP LB + Cloud CDN (if used)
TRUSTED_PROXY_HEADERS=x-forwarded-for,x-forwarded-proto
```

**Azure Application Gateway:**

```bash
TRUSTED_PROXY_COUNT=1
TRUSTED_PROXY_HEADERS=x-forwarded-for,x-forwarded-proto
```

## File Permissions and Security

### Directory Structure and Permissions

```
/opt/wireguard-mesh-manager/
├── backend/
│   ├── data/                    # SQLite database and encrypted keys
│   │   └── wireguard.db         # 600 permissions
│   ├── alembic.ini              # 644 permissions
│   └── migrations/              # 755 permissions
├── frontend/
│   ├── .next/                   # 755 permissions
│   └── public/                  # 755 permissions
└── docker-compose.yml           # 644 permissions
```

### Recommended Permissions

```bash
# Create dedicated user
sudo useradd -r -s /bin/false wmm
sudo groupadd wmm

# Set ownership
sudo chown -R wmm:wmm /opt/wireguard-mesh-manager

# Set permissions
sudo chmod 755 /opt/wireguard-mesh-manager
sudo chmod 755 /opt/wireguard-mesh-manager/backend
sudo chmod 755 /opt/wireguard-mesh-manager/frontend
sudo chmod 700 /opt/wireguard-mesh-manager/backend/data
sudo chmod 600 /opt/wireguard-mesh-manager/backend/data/wireguard.db

# Log files
sudo touch /var/log/wmm-backend.log /var/log/wmm-frontend.log
sudo chown wmm:adm /var/log/wmm-*.log
sudo chmod 640 /var/log/wmm-*.log
```

### systemd Service Configuration

```ini
# /etc/systemd/system/wmm-backend.service
[Unit]
Description=WGM Backend Service
After=network.target
Wants=network.target

[Service]
Type=simple
User=wmm
Group=wmm
WorkingDirectory=/opt/wireguard-mesh-manager/backend
Environment=DATABASE_URL=sqlite:///./data/wireguard.db
Environment=LOG_LEVEL=INFO
Environment=TRUSTED_PROXY_HEADERS=x-forwarded-for,x-forwarded-proto
Environment=TRUSTED_PROXY_COUNT=1
ExecStart=/usr/local/bin/uvicorn main:app --host 127.0.0.1 --port 8000
Restart=always
RestartSec=10

# Security
NoNewPrivileges=true
PrivateTmp=true
ProtectSystem=strict
ReadWritePaths=/opt/wireguard-mesh-manager/backend/data /var/log
ProtectHome=true
RemoveIPC=true

[Install]
WantedBy=multi-user.target
```

### Docker Security Considerations

When using Docker in production:

```yaml
# Production docker-compose.override.yml
version: '3.8'

services:
  backend:
    ports:
      - '127.0.0.1:8000:8000' # Only bind to localhost
    volumes:
      - ./backend/data:/app/data:rw
      - /var/log/wmm:/app/log:rw
    environment:
      - DATABASE_URL=sqlite:///./data/wireguard.db
      - LOG_LEVEL=INFO
      - TRUSTED_PROXY_HEADERS=x-forwarded-for,x-forwarded-proto
      - TRUSTED_PROXY_COUNT=1
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M
        reservations:
          cpus: '0.5'
          memory: 256M

  frontend:
    ports:
      - '127.0.0.1:3000:3000' # Only bind to localhost
    environment:
      - NODE_ENV=production
      - NEXT_PUBLIC_API_URL=https://wireguard.example.com
```

## Network Security

### Firewall Configuration

```bash
# UFW example
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow ssh
sudo ufw allow 80/tcp   # Redirect to HTTPS
sudo ufw allow 443/tcp  # HTTPS
sudo ufw enable
```

### Isolation Strategy

1. **Database**: Only accessible by backend process
2. **Backend/Frontend**: Only accessible via reverse proxy
3. **Admin Access**: SSH key-based authentication only
4. **Monitoring**: Separate network segment for metrics collection

## Environment-Specific Configurations

### Development Environment

- Use `docker-compose.yml` with localhost binding
- Self-signed certificates acceptable for testing
- Relaxed security headers for debugging
- Volume mounts for live code reloading

### Staging Environment

- Production-like security configuration
- Staging-specific subdomain (e.g., staging.wireguard.example.com)
- Separate database instance
- Integration with CI/CD pipeline

### Production Environment

- Full security hardening applied
- High-availability deployment (multiple instances)
- Backup and disaster recovery procedures
- Comprehensive monitoring and alerting

## Monitoring and Observability

### Health Check Endpoints

- Backend: `GET /health` - Service health and database connectivity
- Frontend: `GET /api/health` - Frontend service health
- External monitoring: SSL certificate expiry, response times

### Metrics Collection

- Application metrics via `/metrics` endpoint
- Reverse proxy metrics (Nginx stub_status or Caddy metrics)
- System metrics (CPU, memory, disk usage)
- Security events (failed authentication, rate limit violations)

### Log Management

```bash
# Log rotation configuration
# /etc/logrotate.d/wireguard-mesh-manager
/var/log/wmm-*.log {
    daily
    missingok
    rotate 30
    compress
    delaycompress
    notifempty
    create 640 wmm adm
    postrotate
        systemctl reload wmm-backend
    endscript
}
```

## Backup and Disaster Recovery

### Data Backup Strategy

1. **Database Backup**: Encrypted SQLite backup to secure location
2. **Configuration Backup**: Version-controlled deployment configs
3. **Certificate Backup**: Secure backup of TLS certificates
4. **System State Backup**: Periodic system configuration snapshot

### Recovery Procedures

1. **Service Recovery**: Automatic restart via systemd/health checks
2. **Data Recovery**: Restore encrypted database from backup
3. **Full Recovery**: Re-deploy from infrastructure as code
